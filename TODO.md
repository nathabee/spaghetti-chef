# 0.4.9 — Camera Job Storage Model And Snapshot Ownership

## Status

Planned / active correction.

0.4.8 introduced camera snapshot/archive administration, but the current model is still incoherent because the old `archive` concept was actually used as a duplicated snapshot timeline.

For 0.4.9, the current fake archive concept is removed from the active capture flow.

The active concepts are now:

```text
CameraJob
Source snapshot
Snapshot timeline
```

The future archive/compression feature is not part of capture-time behavior.

---

## Problem being fixed

The current implementation still groups retained camera files by:

```text
snapshot/unassigned/
snapshot/<activePrintJobId>/
```

or similar fallback concepts.

This is not stable enough.

The system needs a real camera/image job id generated by the database.

That camera job id becomes the owner of all retained source snapshots for one camera run.

---

## Correct terminology

```text
Live snapshot
  latest.jpg
  previous.jpg
  delta.jpg

Source snapshot
  A retained image captured by the camera and stored on disk.

Camera job
  One camera/image capture run for one printer.
  It owns source snapshots.
  It may link to a normal print job.
  It is not itself a print job.

Print job
  Existing PrinterHub job used for print execution, SD upload, or controlled commands.

Camera analysis session
  Detector/analysis session.
  It may be linked to a camera job, but it should not own filesystem folders directly.

Delta set
  Future derived image-difference set generated from source snapshots.

Calculation run
  Future spaghetti-detection calculation using one selected delta set.

Archive/compression
  Future explicit admin operation.
  It is not performed automatically during capture.
```

---

This endpoint already exists and must stay stable:

```text

```

```text

```

In the admin API, `snapshot` means retained source snapshot data, not the live latest image endpoint.

## Camera job creation rule

When camera capture starts for a printer:

```text
1. Look for an active camera job for this printer.
2. If an active camera job exists, use it.
3. If none exists, create a new camera job.
4. If an active normal print job exists for this printer, store its id in linked_print_job_id.
5. If no active print job exists, linked_print_job_id is null.
6. The camera job id is generated by the database.
7. Camera jobs must never execute printer commands.
8. Camera jobs must never trigger SD upload, print start, pause, resume, cancel, or printer movement.
```

The first standalone camera capture for a printer may create:

```text
camera_jobs:
  id = 1
  printer_id = printer-1
  linked_print_job_id = null
  state = RUNNING
```

A capture during an active print job may create:

```text
camera_jobs:
  id = 2
  printer_id = printer-1
  linked_print_job_id = <printJobId>
  state = RUNNING
```

## Capture behavior

Capturing a frame must:

```text
1. copy latest.jpg to previous.jpg if latest.jpg exists
2. write the new frame to latest.jpg
3. create or reuse a camera job
4. write one retained source snapshot under snapshot(s)/<cameraJobId>/
5. save one camera snapshot entry linked to camera_job_id
6. apply Retained snapshots inside the selected camera job folder
7. record the camera capture event
```

Capturing a frame must not:

```text
1. write a second archive copy
2. populate archive/
3. save CameraArchiveEntry
4. create duplicate retained files for the same frame
5. group new source snapshots under unassigned as the final model
```

# 0.4.10 — Delta Sets And Calculation Runs

## Status

Planned after 0.4.9.

This phase depends on the CameraJob model.

A delta set is generated from source snapshots of one selected camera job.

A calculation run is generated from one selected delta set.

Do not mix delta generation and spaghetti calculation. They are separate workflows.

---

## Existing settings

The system already has:

```text
Capture interval seconds
Retained snapshots
```

Use these for source snapshot capture.

```

```

Output:

```text

```

---

## Delta snapshot step rule

Delta snapshot step is sampled, not sliding.

Assume chronological snapshots:

```text
S1, S2, S3, S4, S5, S6, S7, S8, S9, S10, S11, ...
```

If `Delta snapshot step = 1`:

```text
S1 -> S2
S2 -> S3
S3 -> S4
S4 -> S5
```

If `Delta snapshot step = 10`:

```text
S1  -> S11
S11 -> S21
S21 -> S31
```

Intermediate snapshots are not used for that delta set.

With 5000 snapshots and step 10, the expected result is about 500 delta frames, not about 4990.

Planned after 0.4.10.

## Replay modes

The admin UI should eventually support three replay modes:

```text
1. Snapshot replay
   Shows raw object creation from source snapshots.

2. Delta replay
   Shows visual difference evolution from a selected delta set.

3. Calculation replay
   Shows delta frames together with numeric detection results.
```

---

## Replay controls

```text
Play
Pause
Stop
Replay display ms
Frame counter
Selected frame preview
Selected metadata panel
```

---

## Compression

Compression is a future explicit admin action.

It is not capture-time behavior.

Compression may delete source snapshots and therefore invalidate derived delta sets and calculation runs.

First safe rule:

```text
If source snapshots are compressed/deleted, delete or mark invalid all dependent delta sets and calculation runs.
```

Admin warning:

```text
This operation deletes source snapshots and may invalidate delta sets and calculation runs. Deleted source snapshots cannot be reconstructed.
```

---

## 0.4.11 acceptance checklist

* Admin can replay source snapshots.
* Admin can replay delta sets.
* Admin can inspect calculation results over time.
* Replay display ms controls playback speed.
* Compression requires confirmation.
* Compression never touches latest.jpg, previous.jpg, or delta.jpg.
* Compression never deletes files outside the selected printer/camera-job storage.
